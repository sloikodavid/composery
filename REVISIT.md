## Patches -> direct edits

These patches target `src/node/*` (our source, not VS Code). They should become
direct edits to the source files so AI agents see the final state, not unpatched
source + a diff. The VS Code patches (targeting `lib/vscode/*`) must stay quilt
because we don't own the VS Code source (it's a submodule).

Patches to convert to direct edits:
- `auth-flow.diff` -> direct edits in src/node/{cli,http,main,routes/*}.ts
- `no-generated-password.diff` -> direct edit in src/node/cli.ts
- `persistence-readiness.diff` -> direct edits in src/node/{persistence/readiness,routes/health,routes/index}.ts
- `browser-friendly-url.diff` -> direct edits in src/node/{main,util}.ts
- `branding.diff` (src/node/* parts only) -> direct edits in src/node/{cli,http,main,util,wrapper}.ts
- `auth-actions.diff` (src/node/cli.ts parts only) -> direct edit in src/node/cli.ts
- `branding.diff` (ci/build/build-vscode.sh parts) -> direct edit in ci/build/build-vscode.sh

After converting these, remove them from `patches/series` and delete the .diff files.
The remaining series would be code-server's 25 VS Code patches + our VS Code patches
only.

## Overlay -> build script integration

Currently the Dockerfile copies `packages/ide/extensions/` and
`packages/ide/workbench-assets/` onto the release after the build. Consider
integrating these into `ci/build/build-release.sh` instead, so the build script
is self-contained:

```bash
# In build-release.sh, add to bundle_vscode():
rsync ../extensions/ "$VSCODE_OUT_PATH/extensions/"
rsync ../workbench-assets/ "$VSCODE_OUT_PATH/out/vs/code/browser/workbench/"
```

Also consider whether `workbench-assets/` could be referenced from VS Code source
(inside the submodule) via the quilt patches, instead of being overlaid after
the build. Currently `overlays.diff` and `fonts.diff` add `<link>` tags to
`workbench.html` that point to these files. The files go into the built output,
not the source. This is inherently an overlay — it can't go in the submodule.

## postinstall.sh and npm

`postinstall.sh` runs `npm ci` in three subdirectories:
- `test/` (jest, playwright deps)
- `test/e2e/extensions/test-extension/`
- `lib/vscode/` (VS Code deps with electron headers)

These stay npm because they have their own `.npmrc` and lockfiles. The main
package uses pnpm (managed by root pnpm-lock.yaml). Consider converting the
subdirectory installs to pnpm for consistency, but `lib/vscode/.npmrc` has
electron-specific config (`disturl`, `target`, `runtime`, `build_from_source`,
`legacy-peer-deps`) that needs pnpm-specific testing.

## Test infrastructure

- code-server's unit tests use jest (configured in package.json). Our root
  uses vitest. These don't conflict — jest runs inside `packages/ide/`, vitest
  runs at the root. Verify jest works under pnpm.
- `test/package.json` has its own `resolutions` field. These need to go into
  `pnpm-workspace.yaml` overrides or stay as-is (npm ci in test/ is separate
  from pnpm).
- `test/` is installed by `postinstall.sh` using `npm ci`, not by pnpm. So
  test/package.json resolutions stay as-is. They only matter if someone runs
  `npm install` instead of `npm ci` in test/.

## VS Code submodule updates

- The submodule is pinned to `9b8ae15a8cf95b9bce1b590b42954530f440e816`
  (VS Code commit used by code-server 4.118.0).
- Updating VS Code: bump the submodule, re-run `quilt push -a`, fix any
  broken patches, rebuild.
- Renovate `git-submodules` manager can automate bump PRs.
- When VS Code updates, the 50 patches (code-server's 25 + our 25) may need
  adjustment. code-server's patches are maintained by the code-server project
  (we'd need to pull their updated patches from the new code-server version).
  Our patches are maintained by us.

## code-server upstream tracking

We've diverged from code-server. We do NOT sync from coder/code-server anymore.
If code-server fixes a bug or adds a feature we want, we read their diff and
apply it manually to our source. We are NOT a fork that rebases on upstream.

However, code-server's own VS Code patches (the 25 in our patches/ directory
that came from code-server) DO get updated when code-server updates. If we want
to pull an updated VS Code patch stack from a newer code-server release, we
replace those 25 files. Our 25 patches stay unchanged.

## Dockerfile pnpm lockfile

The Dockerfile uses `pnpm install` (without `--frozen-lockfile`) because the
Docker context only includes `packages/ide/`, not the root `pnpm-lock.yaml`.
This means dependency versions in the Docker build are resolved from
`package.json` ranges, not pinned by a lockfile.

Options to fix:
1. Generate a standalone `pnpm-lock.yaml` for `packages/ide/` by running
   `pnpm install --lockfile-only --ignore-workspace` and commit it. Then use
   `pnpm install --frozen-lockfile --ignore-workspace` in the Dockerfile.
   Downside: a separate lockfile from the root one.
2. Restructure the Docker build to include the root `pnpm-workspace.yaml` and
   `pnpm-lock.yaml` and set up the full workspace. More complex but aligns with
   the monorepo model.
3. Accept the current state: the base image digest is the reproducibility
   boundary, and dependency ranges in package.json are intentionally conservative.

## Dockerfile VSCODE_COMMIT tracking

The Dockerfile has `ARG VSCODE_COMMIT=9b8ae15a8cf95b9bce1b590b42954530f440e816`
with a `renovate: datasource=git-tags` comment. Renovate's `git-tags` datasource
may not work correctly for VS Code (which uses a complex tagging scheme).
Verify that Renovate can actually bump this, or switch to a custom datasource
similar to the old code-server one.
